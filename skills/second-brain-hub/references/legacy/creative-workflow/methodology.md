# 创造性工作流（思想群岛 + 海明威之桥 + 压缩范围）

## I — 方法论骨架 (Interpretation)

创造性工作流由三个互补策略组成，分别解决前、中、后三个阶段的典型障碍：

1. **思想群岛** — 解决"空白页恐惧"（怎么开始？）：从第二大脑收集已有素材→排列成"岛屿"→搭建逻辑桥梁
2. **海明威之桥** — 解决"重启困难"（下次怎么接上？）：在精力充沛时主动停下，写下下一步+当前状态+易忘细节
3. **压缩范围** — 解决"交付恐惧"（怎么收尾？）：主动砍掉最困难的部分，先交付核心，其余留待后续迭代

> 三大策略的详细解释与核心哲学，参见 [three-strategies-deep-dive.md](references/three-strategies-deep-dive.md)

## A2 — 触发场景 (Future Trigger) ★

### 用户会在什么情境下需要这个 skill?

1. 面对空白文档/PPT/画布，盯着看了半天一个字也写不出来（用思想群岛）
2. 业余项目（写作/编程/设计）每次都虎头蛇尾——开头兴致勃勃，过几天就再也不想打开（用海明威之桥）
3. 接了一个大项目但不知从何入手，觉得工程量太庞大（组合使用三种策略）
4. 完美主义导致永远在"打磨"而不交付——害怕作品被批评（用压缩范围）
5. 碎片时间无法利用——总觉得需要整块时间才能做创意工作（用海明威之桥 + 半熟素材）

### 语言信号 (用户的话里出现这些就应激活)

- "不知道怎么写开头"
- "空白页恐惧"
- "项目太大不知从何下手"
- "总是做一半就不想做了"
- "每次重新打开项目都忘了上次做到哪"
- "怎么把大项目拆小"
- "完美主义让我交不出东西"
- "需要一个创作流程"
- "虎头蛇尾"

### 与相邻 skill 的区分

- 与 `diverge-converge` 的区别：发散聚合是底层理论模型（解释"为什么"），创造性工作流是实操工具集（解决"怎么做"）
- 与 `intermediate-packets` 的区别：半熟素材是创作原材料，创造性工作流是如何使用这些原材料的方法
- 与 `knowledge-lifecycle` 的区别：创造性工作流侧重"创作过程中的技巧"，知识生命周期侧重"项目的启动和收尾标准化"

## E — 可执行步骤 (Execution)

当 skill 被激活后，agent 应按以下步骤执行:

<HARD-GATE id="blocker-diagnosed-before-workflow">
未诊断创作卡点之前，不得直接给大纲、草稿或任务清单。

必须先归类为:
1. 开始困难 → 思想群岛
2. 持续困难 → 海明威之桥
3. 收尾困难 → 压缩范围

禁止的行为:
- 默认用户只是需要大纲
- 同时套用三个策略但不说明主攻方向
- 用通用写作建议替代创作流程
</HARD-GATE>

<HARD-GATE id="next-artifact-before-finish">
结束前必须产出一个可继续推进的中间产物。

至少满足一项:
1. 思想群岛素材清单
2. 海明威之桥下一步记录
3. 最小可交付范围

禁止的行为:
- 只解释方法论不落到产物
- 只鼓励用户继续收集素材
- 没有下一步动作就结束
</HARD-GATE>

1. **诊断用户的创作障碍**
   - 询问：你现在卡在创作的哪个阶段？是不知道如何开始、中途无法持续、还是无法收尾？
   - 完成标准：确定推荐哪个策略作为主攻方向

2. **若卡在"开始"→使用思想群岛**
   - 搜索第二大脑中与该主题相关的所有现有笔记和半熟素材
   - 将它们集中到一个空白笔记中，排列成要点列表（不需要逻辑顺序，先堆上去）
   - 将相关的参考笔记链接附在每个要点后面
   - 然后再做逻辑排序和桥梁搭建
   - 完成标准：用户面前不再是空白页面，而是一个至少有5个要点的素材大纲

3. **若卡在"持续"→搭建海明威之桥**
   - 在当前工作阶段结束前（或在下次重启前回顾上次的状态），写下：
     - 下一步要做什么（具体到可执行的动作）
     - 当前最大的未决问题
     - 任何容易被遗忘的细节
   - 如果项目已中断，引导用户先读这些"桥"再开始工作
   - 完成标准：建立了一个明确的"下次入口"，下次打开项目时30秒内能知道该做什么

4. **若卡在"收尾"→使用压缩范围**
   - 列出项目的所有功能/内容/子任务
   - 用"必要→重要→锦上添花"三档分类
   - 将"重要"和"锦上添花"的条目移入第二大脑保存（而非删除），设定为"后续迭代"
   - 只交付"必要"部分
   - 完成标准：用户已确定了最小可交付范围

## 合理化防御（Prebuttal）

Agent 在压力下会为跳过创意工作流步骤找到合理理由。以下是预先反驳：

| Agent 的借口 | 预先反驳 |
|-------------|---------|
| "用户想法很清晰了，直接写就行" | 清晰的想法也需要"思想群岛"做结构化。跳过素材收集 = 从零开始的恐惧会卷土重来 |
| "这个项目很小，不需要完整流程" | 小项目也会卡住。海明威之桥对任何规模的创作都有用——知道下一步从哪开始，就是不卡死的保证 |
| "用户着急要结果，跳过中间步骤更快" | 跳过步骤现在看起来更快，但半途卡住的花费远超前置投入。创作工作流的价值就是防止卡住 |
| "素材够多了，不需要再收集" | 素材够了≠素材结构化了。思想群岛的核心不是收集更多，是把已有的连接成可导航的群岛 |
| "压缩范围会让内容太少" | 压缩范围是聚焦，不是删减。缩小范围 → 提高深度 → 作品质量提升。小而深 > 大而浅 |
| "这是突发灵感，不用走完整流程" | 灵感最需要结构化。没有流程的灵感 = 写了一半就放弃的草稿 |
