## 二、当前状态审计 — 七大发现

### 🔴 发现 1：description 是"教程"而非"路由器"

**问题**：

```yaml
# 当前 second-brain-hub 的 description (103 字)
description: |
  第二大脑知识管理中枢——所有信息输入的统一入口。接收语音转写、网页链接、
  文件、聊天消息等任意输入,自动进行意图识别,调度对应的CODE方法论skill
  (抓取→组织→提炼→表达),最终将知识写入Obsidian笔记库。
  
  触发场景: 当用户说"记一下""保存这篇文章"...
```

Agent 在触发前只能看到 description,你把完整流程写进去 = 召回时的 token 浪费。

**对标最佳实践**（来自你的笔记《如何写好一个 Agent Skill》）：

> description 只回答两个问题："这个 Skill 做什么"和"什么时候使用它"。不要把完整流程写进 description。

**改进方向**：

```yaml
# ✅ 精简后 (45 字)
description: |
  第二大脑知识管理统一入口 — 记录灵感、保存网页、回顾整理、启动创作。
  触发：用户说"记一下""保存这篇文章""回顾本周""写一篇""找一下"。
  不适用于：与知识管理无关的纯对话、纯 Obsidian 技术问题。
```

**影响面**：Hub + 9 个方法论 Skill 的 description 都需要重写。

### 🔴 发现 2：缺少硬门控 (Hard Gate)

**问题**：

当前 Skill 大量使用软建议，例如 Hub 中：

```markdown
1. 根据 vault_path 拼接 Vault 内的状态文件路径
2. 读取 Vault 内状态文件（与项目配置合并）
   - 若存在 → 读取 last_operations 等运行时数据
   - 若不存在 → 以项目配置为准，自动创建该文件
```

这是"建议先做 A"，Agent 在上下文压力下会跳过。

**对标最佳实践**（来自《Skill系统设计理念与工程实践-淘天苏雄》）：

> 关键路径上不要用"建议先做A"，要用**"没做完 A 不准做 B"**。Agent 在上下文压力下会跳过软约束——门控让它跳不过去。

**改进方向**：

```markdown
<HARD-GATE id="vault-config">
在未确认 vault_path 和 vault_name 之前，不得执行任何 Obsidian 文件写入操作。
若配置为 null，必须先引导用户提供路径并写入 hub-state.json，然后重新进入流程。
</HARD-GATE>

<HARD-GATE id="para-classification">
在调用 para-system 确认目标路径之前，不得创建笔记文件。
未分类的内容禁止直接写入 Vault 根目录。
</HARD-GATE>
```

**影响面**：Hub 至少需要 3 个门控点；高风险 Skill（progressive-summarization、creative-workflow）各需 2 个。

---

### 🔴 发现 3：缺少合理化防御 (Prebuttal)

**问题**：

当前 Skill 没有预见 Agent 的"跳过理由"。

**对标最佳实践**（来自《Skill系统设计理念与工程实践-淘天苏雄》）：

> Agent 在压力下会为跳过规则找到**听起来合理的理由**。Skill 要提前写出这些借口并反驳。

**改进方向**：

在 Hub 的"第三步-外源"场景中加入：

```markdown
### 合理化防御

Agent 可能会说的理由 vs 预先反驳：

| Agent 的借口 | 反驳 |
|-------------|------|
| "这只是一篇普通文章，直接保存就好，不用走 capture-criteria" | capture-criteria 不是为"特殊文章"设计的，而是为所有外源输入设计的标准筛选流程 |
| "用户已经说'保存'了，说明他已经判断过价值" | 用户的"保存"是初步意向，capture-criteria 帮助用户二次确认并避免数字囤积 |
| "我已经理解了 PARA 分类规则，不需要调用 para-system" | 理解 ≠ 执行。para-system 提供的是确定性分类逻辑，不是启发 |
| "这是小修改，不用完整走渐进式摘要" | 每条规则都在这类借口中被绕过。没有"小修改"豁免权 |
```

**影响面**：Hub + 4 个核心方法论 Skill（capture-criteria、para-system、progressive-summarization、creative-workflow）。

### 🟡 发现 4：未实现三层加载分离

**问题**：

当前所有 Skill 的全部内容在触发时一次性加载，没有分离 frontmatter / 正文 / 按需资源。

以 `capture-criteria` 为例，SKILL.md 包含：
- 四大标准的完整解释（启发性/实用性/个性/新奇性）
- 书中三个案例（泰勒·斯威夫特/杰里·森菲尔德/作者情感文件夹）
- 共鸣原则的神经科学解释

这些内容应该放进 `references/capture-theory.md`，只在 Agent 需要时才读取。

**对标最佳实践**（来自《Skill系统设计理念与工程实践-淘天苏雄》）：

| 层 | 位置 | 何时加载 | 放什么 |
|---|------|---------|--------|
| 发现层 | `name` + `description` | 启动时路由 | 触发条件、使用场景 |
| 执行层 | SKILL.md 正文 | 触发后 | 完整流程、核心规则 |
| 按需层 | `scripts/` `references/` `assets/` | 执行中需要时 | 脚本、领域知识、模板 |

**改进方向**：

```
capture-criteria/
├── SKILL.md              # 只保留：流程步骤 + 四大标准清单（不展开解释）
├── references/
│   ├── theory.md         # 神经科学原理、共鸣机制
│   └── examples.md       # 书中三个案例
└── scripts/
    └── quick_check.py    # 可选：批量检查收藏夹的共鸣度
```

**影响面**：14 个 Skill 都需要重构目录结构。预估 token 节省 40%。

### 🟡 发现 5：缺少评测体系

**问题**：

当前项目没有任何 Skill 评测机制。你的知识库中有《Agent评测体系-从Demo到生产可用》，但从未应用到本项目。

**对标最佳实践**（来自《Agent评测体系-从Demo到生产可用》）：

> Agent 评测不是跑分，是把非确定性的智能行为持续收敛为可发布的工程质量。

**核心指标**（针对 Skill）：

| 指标类 | P0（门禁） | P1（对比） |
|--------|:--:|:--:|
| 触发准确性 | ✅ 正确路由 | 边缘case召回 |
| 流程执行完整性 | ✅ 无跳步 | 门控生效 |
| 工具参数正确性 | ✅ 无错误调用 | 降级处理 |
| 产物可用性 | ✅ 笔记可读 | frontmatter完整 |

**改进方向**：

```
tests/
├── hub/
│   ├── test_intent_recognition.json   # 50 条意图识别用例
│   ├── test_scene_routing.json        # 每个场景 10 条
│   └── test_vault_writing.json        # frontmatter 完整性
├── capture-criteria/
│   └── test_resonance_judgment.json   # 共鸣标准判断
└── progressive-summarization/
    └── test_distill_levels.json       # L1-L4 提炼质量
```

配套脚本：

```bash
scripts/
├── eval_skill.py          # 单 Skill 评测
├── eval_batch.py          # 批量评测（n-runs ≥ 3）
└── badcase_rca.py         # 根因分析
```

**影响面**：新增测试基础设施，每个 Skill 至少 20 条用例。

### 🟡 发现 6：检索能力落后

**问题**：

当前依赖 Obsidian 关键词搜索，无索引、无语义检索。你的知识库《对比分析-路人甲方案-vs-我们》中已明确指出这是最大差距：

> 他们："先看目录（index.md），只读跟你这个问题相关的那几页"
> 我们："搜'Agent'返回所有含'Agent'的笔记，无法区分重要性"

**改进方向**（两条路径任选其一）：

**路径 A：Index 索引检索**（借鉴 Karpathy 方案）

```
{vault_path}/
├── index.md              # 库的目录地图
├── topics/
│   ├── AI前沿.index.md
│   ├── 视频创作.index.md
│   └── 知识管理.index.md
```

Agent 搜索时先扫 index，找到相关主题后再精准定位笔记。

**路径 B：语义搜索层**（向量嵌入）

```python
# scripts/semantic_search.py
import chromadb
from sentence_transformers import SentenceTransformer

# 一次性构建索引
def build_index(vault_path):
    client = chromadb.Client()
    collection = client.create_collection("second-brain")
    
    for note in glob(f"{vault_path}/**/*.md"):
        content = read(note)
        embedding = model.encode(content)
        collection.add(embeddings=[embedding], documents=[content], ids=[note])

# 查询时
def search(query, top_k=5):
    results = collection.query(query_texts=[query], n_results=top_k)
    return results['documents']
```

**代价**：路径 A 轻量但需手动维护；路径 B 自动化但需 `pip install chromadb sentence-transformers`。

**建议**：P1 先做路径 A（1天），P2 升级到路径 B（2天）。

### 🟢 发现 7：缺少"对话保存"场景

**问题**：

你的知识库《对比分析-路人甲方案-vs-我们》中提到：

> 跟 AI 聊完天，觉得聊出了一些好东西，说一句「帮我把这段对话存进第二大脑」，AI 就把原话归档进 raw/，同时提炼成一页带链接的知识页。这个场景我们的 Hub 没有覆盖。

**改进方向**：

在 Hub 新增**场景 8：对话归档**

```markdown
## 第三步-对话归档

触发条件：用户说"把这段对话存下来""保存刚才的讨论""归档这次聊天"

流程：
1. 提取当前会话的关键轮次（用户可指定范围，如"最近5轮"）
2. 调用 capture-criteria 判断对话中哪些部分值得保存
3. 双轨写入：
   - raw/ 目录：完整对话原文（时间戳 + 逐轮记录）
   - wiki/ 目录：提炼后的知识页（调用 progressive-summarization L2 提炼）
4. 自动挂载到相关主题下（wikilinks）
```

**代价**：1 天开发 + 对话历史提取接口（取决于 Coze 平台能力）。

## 四、风险与权衡

### 风险 1：三层加载分离可能破坏现有流程

**缓解措施**：
- 先做 1 个 Skill 试点（建议 capture-criteria），验证后再批量迁移
- 保留 SKILL.md 的核心流程骨架，只把"理论解释"和"案例"移到 references/

### 风险 2：评测体系建设成本高

**缓解措施**：
- P1 只做 Golden Set（50 条核心用例）
- P2 再扩展到边缘 case 和 badcase 回流
- 用 LLM-as-Judge 辅助生成用例变体

### 风险 3：语义搜索依赖外部库

**缓解措施**：
- P1 先做 Index 索引（零依赖）
- P2 再可选升级语义搜索
- 两套方案并存，Index 作为 fallback
