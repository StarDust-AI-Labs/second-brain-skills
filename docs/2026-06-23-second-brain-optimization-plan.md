# 第二大脑项目扫描与优化方案

- 日期: 2026-06-23
- 范围: 当前仓库结构、核心设计文档、`.agents/skills` 与 `.claude/skills` 下的 Skill 体系、Hub 设计与测试提示
- 目标: 先把第二大脑从“设计完整”推进到“运行可验证、状态可维护、迭代可持续”

---

## 1. 当前项目画像

这个项目不是单纯的笔记模板库，而是一套运行在 Claude/Codex 环境里的个人知识管理 Skill 生态。它的核心目标是把“信息录入”变成“知识加工与创作产出”，并通过 Obsidian Vault 做长期存储。

当前体系可以概括为五层架构:

1. 输入层: 语音转写、网页链接、文件、聊天消息等任意信息入口。
2. Agent 层: `second-brain-hub` 作为统一中枢，负责意图识别、场景路由、上下文状态和统一写入。
3. 方法论层: 9 个来自《打造第二大脑》的方法论 Skill，围绕 CODE 流程组织。
4. 工具层: 5 个 Obsidian 工具 Skill，负责 Markdown、CLI、Bases、Canvas、网页提取等能力。
5. 存储层: Obsidian Vault，通过 PARA 目录、frontmatter、状态文件和画布承载结果。

核心理念非常清晰: 信息管理的终点不是“知道”，而是“做出”。这也是后续优化时最应该保留的北极星。

---

## 2. 当前版本状态

从 README、Hub SKILL 和计划文档看，项目宣称已经完成:

- MVP: Vault PARA 重组、Hub 中枢、灵感速记、保存外源。
- v1.1: 收件箱批量处理、创作启动、渐进式归纳深度集成。

实际扫描到的现状:

- `.agents/skills` 下存在完整 Skill 体系，且当前 Codex 环境正在识别这些 Skill。
- `.claude/skills` 下也存在一套 Skill，说明项目存在“双 Skill 源”的历史遗留。
- `.claude/hub-state.json` 存在，并包含 `vault_path`、`vault_name`、`active_projects`、自动提炼偏好等 v1.1 状态。
- `second-brain-hub/SKILL.md` 里要求读取 `{项目根目录}/.Codex/hub-state.json`，但当前仓库没有 `.Codex` 目录。
- `docs/superpowers/plans/2026-06-17-second-brain-hub-v1.1.md` 只有一句完成说明，缺少验收证据、变更清单和回归测试记录。
- `test-prompts.json` 已扩展到 30 条，覆盖了主要意图和边界，但目前仍是“提示样例”，还没有自动化验证机制。

结论: 项目方法论与结构设计已经比较成熟，但运行态配置、文档一致性、验证闭环还不够稳。下一步不宜马上扩展更多功能，应该先做一次“系统硬化”。

---

## 3. 主要发现与优化建议

### 3.1 统一运行时配置入口

发现:
- Hub 文档要求 `.Codex/hub-state.json`。
- 仓库实际存在 `.claude/hub-state.json`。
- README 还在描述 `.claude/skills`，但当前可用 Skill 来源是 `.agents/skills`。

建议:
- 明确一个权威配置路径。建议在当前 Codex 项目中使用 `.agents/state/hub-state.json` 或继续保留 `.claude/hub-state.json`，但 Hub SKILL、README、计划文档必须一致。
- 如果需要兼容历史路径，可以定义读取优先级:
  1. `.agents/state/hub-state.json`
  2. `.claude/hub-state.json`
  3. `.Codex/hub-state.json` legacy
- 在 Hub SKILL 中写清楚: 找到多个状态文件时以哪个为准，是否同步写回。

理由:
- Hub 是唯一入口，状态文件是它的“短期记忆”。路径漂移会导致首次配置、Vault 定位、active_projects、自动提炼规则全部失效。

优先级: P0

### 3.2 消除 Skill 双源漂移

发现:
- `.agents/skills` 与 `.claude/skills` 同时存在。
- 二者都包含 `second-brain-hub` 和 9 个方法论 Skill。
- 后续修改如果只改一边，很容易出现“我以为改了，但运行时没生效”的问题。

建议:
- 选定 `.agents/skills` 为当前 Codex 运行时权威源。
- `.claude/skills` 标记为 legacy mirror，或建立同步脚本/说明，明确不能手动双改。
- README 的项目结构改为当前真实路径。
- 对每个 Skill 增加 `version` 和 `last_updated` 字段，便于对比漂移。

理由:
- 第二大脑系统的核心不是代码，而是 Skill 指令本身。Skill 指令漂移就是系统行为漂移。

优先级: P0

### 3.3 给 Hub 增加“可验证运行手册”

发现:
- Hub 的 SKILL.md 很完整，但更像“操作规程”，缺少可执行验收脚本。
- `test-prompts.json` 有 30 条样例，但没有自动检查意图分类、调用链、降级行为是否符合预期。

建议:
- 新增 `docs/runbooks/hub-e2e-validation.md`，记录人工端到端验收流程。
- 新增一个轻量测试脚本，例如 `scripts/validate-test-prompts.ps1` 或 `scripts/validate-test-prompts.js`:
  - 校验 JSON 格式。
  - 检查每条用例包含 `id`、`scene`、`input`、`expected_intent` 或 `expected_behavior`。
  - 输出覆盖率统计: 灵感速记、保存外源、提炼、创作、收件箱、回顾、查询、边界。
- 在每次修改 Hub 后，把测试结果写入 `docs/superpowers/reports/`。

理由:
- 当前测试只能“人脑看起来合理”。下一步应让项目能回答: 修改 Hub 后，有没有破坏已有场景？

优先级: P0

### 3.4 把 v1.1 完成说明补成验收报告

发现:
- v1.1 计划文件只有标题与一句话，缺少完成范围。

建议:
- 补充 `docs/superpowers/reports/2026-06-17-second-brain-hub-v1.1-acceptance.md`，包含:
  - 变更摘要。
  - 涉及文件。
  - 新增/修改的测试用例。
  - 已验证场景。
  - 未验证或待补风险。

理由:
- 第二大脑项目会长期迭代。每个版本需要留下“为什么这样改、验证到什么程度”的记忆，否则未来会反复重新理解旧决策。

优先级: P1

### 3.5 强化 Vault 写入安全与回滚策略

发现:
- Hub 已定义 Obsidian CLI 不可用时直接写 `.md` 文件的降级策略。
- 但还没有看到对批量移动、收件箱清理、删除/移入 `.trash` 的 dry-run、备份和回滚规范。

建议:
- 对所有批量操作增加三段式流程:
  1. preview: 只列出将要移动/修改/删除的文件。
  2. confirm: 用户确认后执行。
  3. report: 生成操作报告，写入 `last_operations`。
- 删除操作默认改为移动到 `📂 存档/_待删除/` 或 `.trash`，不要直接删除。
- 对 frontmatter 批量修改提供变更前后摘要。

理由:
- 第二大脑承载的是长期记忆。批量操作的安全性优先级高于效率。

优先级: P1

### 3.6 设计状态文件 Schema 与迁移规则

发现:
- `.claude/hub-state.json` 已经是 v1.1，并包含自动提炼开关等偏好。
- Hub SKILL 文档里的示例状态与实际状态已经不完全一致。

建议:
- 新增 `docs/reference/hub-state-schema.md`:
  - 字段定义。
  - 必填/可选。
  - 默认值。
  - 版本迁移策略。
  - 敏感路径是否允许提交。
- 每次新增字段时同步更新 schema。

理由:
- 状态文件会成为系统“长期偏好”的载体。没有 schema，后续自动化、仪表盘、回顾任务都会依赖不稳定字段。

优先级: P1

### 3.7 将 v1.2 从“功能列表”拆成可交付任务

当前 v1.2 计划:
- 周月回顾。
- 12 问题过滤。
- Obsidian Bases 仪表盘。

建议拆解:

1. 周回顾最小版
   - 输入: 手动触发“本周回顾”。
   - 读取: 收件箱数量、本周新增笔记、active_projects 最近操作。
   - 输出: 一篇周回顾 Markdown。
   - 验收: 能生成包含“本周核心、收件箱、活跃项目、需关注”的笔记。

2. 12 问题过滤最小版
   - 输入: 用户的 12 个问题清单。
   - 存储: 写入 hub-state 或单独 `twelve-problems.md`。
   - 输出: 保存外源或查询时标注 `related_problems`。
   - 验收: 至少 3 条测试提示能匹配到问题编号。

3. Bases 仪表盘最小版
   - 项目看板: active_projects、最近更新、笔记数。
   - 收件箱监控: inbox_count、超过阈值提醒。
   - 提炼视图: `distill_level < 2` 且最近被访问的笔记。

理由:
- v1.2 的价值不在“功能多”，而在让第二大脑能自我观测、自我提醒。

优先级: P1-P2

### 3.8 建立“录入到表达”的闭环指标

发现:
- 项目理念强调 Express，但当前更多能力集中在 Capture、Organize、Distill。

建议:
- 在 Hub 反馈和周回顾中增加少量指标:
  - 本周新增笔记数。
  - 本周提炼到 L2/L3 的笔记数。
  - 本周由笔记生成的产出数: 文章、大纲、PPT、方案、脚本。
  - 长期未表达的高价值素材。
- 对创作启动场景要求必写“海明威之桥”，并写回 `active_bridges`。

理由:
- 如果没有 Express 指标，系统容易退化成更精致的收藏夹。

优先级: P2

### 3.9 改善外源保存的来源质量与版权边界

发现:
- 保存外源流程会提取网页正文并写入 Obsidian。
- 当前流程关注提炼，但缺少来源质量、版权边界和引用摘要规范。

建议:
- 外源笔记默认保存:
  - URL。
  - 标题。
  - 作者/站点。
  - 抓取日期。
  - 一句话摘要。
  - 少量关键摘录。
  - 自己的启发/下一步用途。
- 对长文全文保存采用“摘要 + 原链接 + 少量摘录”优先，除非用户明确需要完整离线归档。

理由:
- 第二大脑的价值是可行动洞察，不是复制互联网。这样也能降低无意义堆积。

优先级: P2

### 3.10 给项目自身建立维护节奏

建议:
- 每次迭代结束写一篇 `docs/superpowers/reports/YYYY-MM-DD-xxx.md`。
- 每月做一次“第二大脑系统自检”:
  - Skill 源是否漂移。
  - README 是否与真实路径一致。
  - Hub 状态 schema 是否同步。
  - test-prompts 覆盖是否下降。
  - 最近 30 天是否真的产出作品。

理由:
- 这个项目本身也是第二大脑的一部分。它需要被纳入自己的知识生命周期。

优先级: P2

## 4. 推荐迭代路线

### Phase 0: 系统硬化

目标: 让当前 v1.1 可信、可运行、可验证。

任务:
- 统一 Hub 状态文件路径。
- 明确 `.agents/skills` 与 `.claude/skills` 的权威关系。
- 修正 README 中过时的路径描述。
- 新增 Hub 测试提示校验脚本或人工验收 runbook。
- 补 v1.1 验收报告。

验收标准:
- 新人只看 README + Hub SKILL，就能知道状态文件在哪里、Skill 从哪里加载、Vault 如何定位。
- 运行测试校验后能看到 30 条提示的覆盖统计。

### Phase 1: v1.2 最小可用

目标: 让系统开始具备自我观察能力。

任务:
- 实现手动周回顾。
- 建立 12 个兴趣问题的存储与匹配流程。
- 创建 Obsidian Bases 项目看板与收件箱视图。

验收标准:
- 用户说“本周回顾”，系统能生成结构化回顾笔记。
- 用户保存外源时，能标注相关问题编号。
- Obsidian 中能看到项目、收件箱、提炼状态的基础仪表盘。

### Phase 2: 表达闭环增强

目标: 让第二大脑更主动地服务创作。

任务:
- `active_bridges` 真正写入与读取。
- 创作启动时自动聚合相关素材，并生成“素材清单/大纲/草稿/下一步”四件套。
- 周回顾中识别“高价值但未表达”的素材。

验收标准:
- 每次创作停止前都有清晰的下次继续点。
- 周回顾能主动推荐 1-3 个可转化为作品的素材包。

### Phase 3: 自动化与提醒

目标: 从手动触发走向低风险自动化。

任务:
- 定时周回顾。
- 收件箱超过阈值提醒。
- 项目停滞检测。
- 批量操作 dry-run + 操作报告。

验收标准:
- 自动化不会直接破坏或移动笔记，只生成报告和建议。
- 用户确认后才执行批量变更。

---

## 5. 建议优先处理的前 5 个任务

1. 确定并修复 Hub 状态文件路径漂移。
2. 确定 `.agents/skills` 为权威 Skill 源，并在 README 中更新。
3. 为 `test-prompts.json` 增加校验脚本或人工验收 runbook。
4. 补写 v1.1 验收报告。
5. 编写 `hub-state` schema 文档。

这 5 件事做完之后，再推进 v1.2 会稳很多。

---

## 6. 一个原则

后续优化不应该追求“让系统更复杂”，而应该追求:

> 每次录入都更轻，每次查找都更准，每次回顾都更短，每次创作都更容易开始。

只要这四个指标在变好，第二大脑就在变聪明。
