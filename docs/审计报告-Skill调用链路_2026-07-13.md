## 二、六大场景的方法论 Skill 调用链路

### 场景全景

```
用户输入
  │
  ▼
🧭 second-brain-hub (意图识别 → 场景路由)
  │
  ├── 场景1 🔖 灵感速记
  │     Hub追问归属 → [capture-criteria ⚡可选] → obsidian-cli 写入
  │
  ├── 场景2 📄 保存外源
  │     defuddle → capture-criteria → para-system → progressive-summarization(L1) → obsidian-markdown → obsidian-cli
  │
  ├── 场景3 ✂️ 提炼加工
  │     obsidian-cli read → progressive-summarization(L1~L4) → obsidian-cli edit
  │
  ├── 场景4 ✍️ 创作启动
  │     intermediate-packets → obsidian-cli search → progressive-summarization(L2) → creative-workflow → obsidian-cli create
  │
  ├── 场景5A 📥 收件箱批量处理
  │     obsidian-cli list → [capture-criteria 批量建议] → para-system → obsidian-cli move
  │
  ├── 场景5 📊 回顾整理
  │     knowledge-lifecycle → obsidian-cli list/search → obsidian-cli create
  │
  └── 场景6 🔍 探索查询
        obsidian-cli search → [twelve-favorite-problems ⚡可选]
```

> ✅ = 必选调用 | ⚡ = 条件调用 | — = 不调用

### 场景1：灵感速记

| 步骤 | 调用 Skill | 作用 | 必选/可选 |
|:--:|------|------|:--:|

**设计依据**：灵感自带"个性"标准（自然满足共鸣），通常跳过 capture-criteria。仅当用户表示"不确定是否值得记"时才触发筛选。

## 四、关键发现

### 发现 1：调用频次严重不均

```
高频（3 次）：
  progressive-summarization  ████████████

中频（2~3 次）：
  capture-criteria           ████████
  para-system                ████████

低频（1 次）：
  intermediate-packets       ████
  creative-workflow          ████
  knowledge-lifecycle        ████

条件调用（0~1 次）：
  twelve-favorite-problems   ██

未被调用（0 次）：
  second-brain-code          —
  diverge-converge           —
```

**progressive-summarization 是唯一被 3 个场景复用的 Skill**，说明"提炼"是横跨多个场景的通用能力。这与 CODE 的设计一致——提炼（Distill）位于信息流的中游，上承组织，下启表达。

---

### 发现 2：2 个 Skill 未被任何场景直接调用 ⚠️

#### `second-brain-code`（CODE 信管法则）

- **定位**：顶层路线图——解释 CODE 四阶段的全局框架
- **未被调用的原因**：Hub 本身就是 CODE 的执行者，6 大场景是 CODE 的场景化拆解
- **CODE ↔ 场景映射**：
  - C（抓取）→ 场景 1 灵感速记 + 场景 2 保存外源
  - O（组织）→ 场景 5A 收件箱处理
  - D（提炼）→ 场景 3 提炼加工
  - E（表达）→ 场景 4 创作启动
- **实际触发方式**：用户主动咨询时才激活——"我该怎么建立知识管理系统""信息太多不知道怎么整理"
- **重构建议**：保持为"诊断引导型"Skill，不纳入自动化链路

#### `diverge-converge`（发散与聚合）

- **定位**：底层创作节律——解释"为什么卡住"
- **未被调用的原因**：其实操对应物是 `creative-workflow`——思想群岛 = 发散，压缩范围 = 聚合。Hub 直接调用 creative-workflow 而非 diverge-converge
- **实际触发方式**：用户表达创作困惑时激活——"素材够了但写不出来""一直在收集但从没产出""完美主义让我交不出东西"
- **重构建议**：可作为场景 4 的**可选前置诊断层**：先调 diverge-converge 诊断卡点类型，再调 creative-workflow 给出操作方案

---

### 发现 3：场景 4（创作启动）是唯一调用 3 个方法论 Skill 的场景

场景 4 调用 intermediate-packets → progressive-summarization → creative-workflow，形成"素材→消化→组装"的创作流水线。但它也是**唯一调用 intermediate-packets 和 creative-workflow 的场景**——这意味着这两个 Skill 的投入产出比最低（每个 Skill 只为 1 个场景服务）。

---

### 发现 4：twelve-favorite-problems 始终是可选的

不论在场景 6（探索查询）— 只在用户定义了问题清单时触发。这符合其设计哲学："用好奇心导航信息消费"——它是一排探照灯，不是一条流水线。

---

### 发现 5：场景 2 是唯一走完整 CODE 流程的场景

6 步全链路（defuddle → capture-criteria → para-system → progressive-summarization → obsidian-markdown → obsidian-cli），覆盖 CODE 的前三步（C→O→D）。这是 Hub 设计中最"重"的路径，也是防御体系最复杂的路径——Hub 为此设置了 4 类合理化防御（跳过 capture-criteria / 跳过 para-system / 跳过 progressive-summarization / 跳过模板）。

---

## 五、方法论 Skill 的互补关系网

除了 Hub 的显式调度，9 个 Skill 之间存在**文档级关联关系**（通过 `related_skills` 和 `composes-with` 声明）：

```
second-brain-code ──composes-with──→ capture-criteria, para-system, progressive-summarization, intermediate-packets, creative-workflow, twelve-favorite-problems, knowledge-lifecycle, diverge-converge
                                     ↑ 唯一的"全关联"Skill——它是所有其他 Skill 的总纲

diverge-converge ──composes-with──→ second-brain-code, creative-workflow, capture-criteria
                                     ↑ 为 creative-workflow 提供理论基础

capture-criteria ──composes-with──→ twelve-favorite-problems, progressive-summarization, second-brain-code, diverge-converge
                                     ↑ "方向性过滤"（12问题）+ "品质性过滤"（4标准）双重叠加

para-system ──composes-with───────→ second-brain-code, intermediate-packets, knowledge-lifecycle
                                     ↑ 为 intermediate-packets 提供存储骨架，为 knowledge-lifecycle 提供维护对象

progressive-summarization ──composes-with──→ second-brain-code, capture-criteria, intermediate-packets
                                             ↑ 其产物（高亮笔记、纲要）是 intermediate-packets 最常见的原材料

intermediate-packets ──composes-with──→ creative-workflow, knowledge-lifecycle, second-brain-code, para-system, progressive-summarization
                                        ↑ 依赖 para-system 存储 + progressive-summarization 提炼产物

creative-workflow ──depends-on─────→ diverge-converge, intermediate-packets
                   ──composes-with──→ knowledge-lifecycle, second-brain-code
                                       ↑ 唯一显式声明 depends-on 的 Skill

knowledge-lifecycle ──depends-on───→ para-system
                    ──composes-with──→ second-brain-code, intermediate-packets, creative-workflow
```

## 六、重构方向建议

### 维度 1：调用链合理性

| 问题 | 现状 | 建议 |
|------|------|------|
| two orphans | second-brain-code / diverge-converge 未被调用 | 保持为"诊断引导型"Skill，不强行插入链路；但在 Hub 中增加"专家模式"触发词 |
| 场景 4 独占 2 个 Skill | creative-workflow + intermediate-packets 只有场景 4 调用 | 考虑将两者合并为一个 `creative-workflow` Skill（half-baked + workflow 一体），或保持独立以减少单文件复杂度 |
| capture-criteria 在 4 个位置出现 | 场景1(可选)+场景2(必选)+场景5A(可选) | 已经是最合理分布——capture-criteria 作为"信息守门人"就应该在多个入口复用 |

### 维度 2：Skill 粒度过细

| Skill | 场景覆盖 | 评价 |
|-------|:--:|------|
| intermediate-packets | 1/7 | 可否并入 creative-workflow？ |
| twelve-favorite-problems | 0+1/7 | 可否降级为 capture-criteria 的 references/？ |

### 维度 3：缺失的调用关系

| 缺失 | 说明 |
|------|------|
| diverge-converge → 场景 4 | 在创作启动前增加可选诊断步骤："要不要先判断一下你卡在发散还是聚合？" |
| second-brain-code → 意图识别 | 当用户意图模糊且追问也得不到明确答案时，用 CODE 框架帮用户定位问题环节 |

## 八、复核结论与重构建议（2026-07-14）

### 8.1 审核结论

这份报告适合作为调用关系的初版地图，但**不能直接作为当前实现的重构基线**。复核顶层 `skills/` 规范源、Hub 的 `required_chain`、测试提示和运行手册后，发现报告没有纳入 2026-07-13 新增的 Hub Run Ledger，并且存在若干链路、场景计数和验证口径不一致的问题。

本次复核的证据范围：

- 规范源：`skills/second-brain-hub/SKILL.md`
- 结构校验：`scripts/validate-test-prompts.ps1`，已通过，覆盖 30 条提示用例
- 行为评测：`scripts/eval_skill.py` 未能运行，原因是当前环境没有可用的 `python` 命令；因此不能把结构校验通过等同于运行时行为已验证

### 8.2 需要修正的审计结论

| 优先级 | 复核发现 | 依据 | 应如何修正报告 |
|---|---|---|---|
| P0 | 场景计数混用 | Hub 有 7 类可路由意图，实际执行流为场景 1、2、3、4、5A、5、6 共 7 条；5A 是收件箱意图的专用子流程 | 明确区分“7 类用户意图”和“7 条执行流”，不要再称为“6 大输入场景” |
| P0 | 灵感速记链路不完整 | Hub 契约要求 `hub.target_routing -> obsidian-markdown -> obsidian-cli/write`，而报告只写到 `obsidian-cli` | 场景 1 补上快速归属与 Markdown 模板步骤；`capture-criteria` 保持条件调用 |
| P0 | 创作启动存在双重真相 | Hub 的 `required_chain` 和测试要求 L2 提炼，但场景 4 的详细流程正文直接从素材检索跳到思想群岛，漏写 `progressive-summarization(L2)` | 先统一 Hub 内部契约，再更新报告。若素材已达到 L2，应将“跳过”显式记录为 `optional_steps_skipped`，不能静默省略 |
| P0 | 回顾整理链路不完整 | Hub 契约包含 `obsidian-markdown -> obsidian-cli/create`，报告只写 `obsidian-cli create` | 场景 5 补上标准回顾笔记渲染步骤 |
| P1 | 收件箱批量处理的 `capture-criteria` 定位不清 | 它不在场景 5A 的必选 `required_chain`，仅在批量建议时可选使用 | 在图和矩阵中标为“批量建议的可选辅助”，不要将它表示成固定层级 |
| P1 | “场景 2 是完整 CODE 链路”表述不准确 | 该链路覆盖 C/O/D，但不包含 E；报告自身也承认表达留给后续创作 | 改为“最完整的保存写入链路”或“C/O/D 完整链路”，避免称为完整 CODE |
| P1 | 关系网遗漏真实依赖 | `intermediate-packets` 实际依赖 `para-system` 与 `progressive-summarization`，报告仅列出其 `composes-with` 关系 | 关系图同时展示 `depends-on`、`composes-with` 与 Hub 运行时调用，三者不能混为一类 |
| P1 | 测试并非链路契约测试 | 30 条提示用例目前主要验证 JSON 结构；部分 `expected_chain` 漏掉 `obsidian-markdown`，且存在与 Hub 契约不完全一致的预期 | 报告应将“测试通过”降级表述为“用例结构通过”，并把链路一致性列为待补验证 |

### 8.3 不建议按覆盖数合并 Skill

`second-brain-code`、`diverge-converge` 和 `twelve-favorite-problems` 的低频调用并不等于冗余：

- `second-brain-code` 是系统诊断与方法选择的总纲，应在用户询问“信息为什么越管越乱”“该从 CODE 哪一步开始”时触发。
- `diverge-converge` 是创作阻塞诊断器，应在“持续收集但无法开始”“方向太多难以收束”“项目迟迟不能交付”等信号出现时，作为场景 4 的可选前置分支触发。
- `twelve-favorite-problems` 保存的是用户的长期注意力方向；它与 `capture-criteria` 的内容质量筛选职责不同，不能降级为后者的参考文件。

同理，`intermediate-packets` 与 `creative-workflow` 应保持分离：前者负责把素材变为可复用资产，后者负责把素材组织为当前产物。二者复用面窄是当前路由不足，而不是合并依据。可以在项目收尾和周回顾中增加“识别可复用半熟素材”的条件分支，提升 `intermediate-packets` 的复用率。

### 8.4 推荐的目标结构

将 Skill 体系按运行职责重组，而不是按调用次数排序：

```text
Agent / Hub 层：second-brain-hub（路由、执行锁、Run Ledger）
└─ SKILL 层
   ├─ 抓取：capture-criteria / twelve-favorite-problems
   ├─ 组织：para-system
   ├─ 提炼：progressive-summarization
   ├─ 表达：intermediate-packets / creative-workflow
   ├─ 工具：defuddle / obsidian-markdown / obsidian-cli / bases / canvas
   └─ 系统维护与诊断：knowledge-lifecycle / code-diagnosis / diverge-converge
```

其中，全局前置条件不应混入每个场景的业务 `required_chain`。建议改为两层结构：

1. `preflight`：所有场景共用的 Vault 配置、意图分类与写入权限。
2. `scenario_chain`：只描述该场景独有的业务步骤和可选分支。

这样可以消除当前“场景 1 写了 `hub.load_config`，其他场景表中却没写，但执行锁又要求所有场景都完成”的歧义。

### 8.5 分阶段重构清单

#### P0：先消除契约漂移

1. 以顶层 `skills/` 为唯一规范源，并修正运行手册中将 `.agents/skills/` 标为“当前权威源”的表述；`.agents`、`.claude` 仅是安装镜像。
2. 建立一个机器可读的 `route-contracts.json`，集中描述每个场景的 `preflight`、必选步骤、可选步骤、输出凭证、写入动作和失败分支。
3. 从该契约生成或校验 Hub 的场景表、`test-prompts.json` 和本审计报告，避免三份手工文本继续漂移。
4. 统一场景 4 的 L2 提炼规则，并补齐场景 1、场景 5 的 `obsidian-markdown` 契约。
5. 对“素材已达 L2”“批量模式未使用 capture-criteria”等分支，要求写入 `optional_steps_skipped` 与理由。

#### P1：把“描述性 Skill”变成可验证能力

1. 为每个被 Hub 调用的 Skill 增加最小能力契约：`inputs`、`outputs`、`gates`、`failure_mode`、`side_effects`。
2. 将测试分成三类：意图路由、链路契约、端到端行为。现有 JSON 结构校验只能覆盖第一类的部分输入完整性。
3. 新增链路一致性检查：每条 `expected_chain` 必须与 `route-contracts.json` 一致；写入场景必须包含模板渲染步骤。
4. 在创作启动中增加显式诊断分支：仅在检测到发散/聚合卡点时调用 `diverge-converge`，不为普通创作额外增加一步。

#### P2：提升复用和可观测性

1. 在周回顾、项目收尾时条件调用 `intermediate-packets`，将可复用素材输出为明确的素材包，而非只留在项目目录。
2. 将 Run Ledger 的完成步骤、跳过理由、失败原因写入可查询的审计记录；用户默认只看结果卡片，需要时再展开技术审计。
3. 为每条路由记录版本号和规范源提交号，使后续审计可以说明“审计的是哪一版契约”。

### 8.6 验收标准

完成 P0 后，以下条件应同时成立：

- `skills/`、安装镜像和运行手册的权威源声明一致。
- Hub 场景正文、`required_chain`、测试预期、审计图四处对同一场景给出相同的必选步骤。
- 所有写入场景都明确模板渲染、目标路径和写入授权。
- 可选步骤被跳过时有可审计理由，而不是从链路中静默消失。
- 结构校验、链路契约校验和至少一轮人工端到端验收分别记录结果。

### 8.8 架构重构实施（2026-07-14）

根据本报告对低频 Skill、契约漂移和公开触发竞争的分析，运行时架构调整为：

```text
输入层
└─ Agent / Hub 层：second-brain-hub（唯一公开入口）
   └─ SKILL 层
      ├─ 抓取 / 组织 / 提炼 / 表达：内部方法论模块
      ├─ 工具：独立 Tool Skill 实现
      └─ 系统维护与诊断：内部维护、CODE 与创作诊断模块
         └─ 存储层：Obsidian Vault
```

实施原则：

1. 平级方法论目录不再包含可发现的 `SKILL.md`，避免绕过 Hub 或与 Hub 竞争触发。
2. 原方法论内容没有删除，迁入 `references/legacy/` 并将入口文件改名为 `methodology.md`。
3. 执行所需的最小规则、硬门控和输出契约提炼为 `module-*.md`，由工作流按需读取。
4. 新增 `diagnosis` 路由，将 CODE 总纲与发散/聚合模型转为可验证的诊断能力；该路由不要求 Vault 配置，也不产生副作用。
5. 周月回顾和项目收尾增加 `intermediate-packets` 条件分支，使可复用素材识别不再只服务创作启动。

重构后的衡量口径从“独立 Skill 被调用几次”改为“内部能力是否在正确条件下执行、是否产生契约输出、跳过是否有证据”。

### 8.9 可移植安装包重构（2026-07-14）

为消除不同 Agent 对嵌套 Skill 发现方式不一致的问题，五个 Obsidian 工具 Skill 已从上游源码包中扁平化到顶层 `skills/`：

```text
skills/
├─ second-brain-hub/
├─ defuddle/
├─ obsidian-markdown/
├─ obsidian-cli/
├─ obsidian-bases/
└─ json-canvas/
```

上游 `LICENSE`、README 和插件市场元数据迁入 `third-party/obsidian-skills-main/`，仅用于许可证与来源归档，不属于运行时安装内容。

`capability-contracts.json` 的实现定位由仓库路径改为两种可移植 locator：

- `{"type": "reference", "path": "references/..."}`：相对于 `second-brain-hub` Skill 根目录解析。
- `{"type": "skill", "name": "obsidian-cli"}`：按目标 Agent 已安装的 Skill 名称解析。

因此用户只需复制顶层 `skills/` 下的六个目录，不依赖仓库外层 `scripts/`、`tests/`、`docs/`、`books/` 或 Python 环境。
