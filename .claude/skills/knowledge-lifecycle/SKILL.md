# 知识生命周期管理（项目清单 + 定期回顾 + 日常维护）

## R — 原文 (Reading)

> "每个项目的启动和收尾时刻……是两个关键节点，或者说两个分岔路口，这也正是你可以做出改变的机会窗口。前者对应的是'项目启动清单'，而后者则对应着'项目完成清单'。"
>
> —— 蒂亚戈·福特，《打造第二大脑》第九章

> "每周小结：1. 清理电子邮件收件箱；2. 检查日历；3. 清理电脑桌面；4. 清理笔记收件箱；5. 确定下周优先事项。"
>
> —— 《打造第二大脑》第九章

> "事实上，任何一种必须做到极致完美才能可靠运行的系统，都会存在严重的内部缺陷。不少'看上去很完美'的系统……分明是脆弱不堪，只要稍不留神便会崩溃。"
>
> —— 《打造第二大脑》第九章

## I — 方法论骨架 (Interpretation)

知识生命周期管理由三个层次的习惯构成，分别在不同时间尺度上维护你的第二大脑健康运转：

### 层次1：项目启动清单 + 项目完成清单（项目尺度）

**项目启动清单**（20-30分钟的"飞行预检"）：
1. 记录有关该项目的所有初始想法（头脑风暴）
2. 检视PARA各类文件夹中可能包含相关笔记的内容
3. 全盘搜索相关术语（确保不遗漏藏在角落的素材）
4. 将找到的所有相关笔记集中到新项目文件夹
5. 撰写笔记大纲和项目方案（思想群岛）

**项目完成清单**（5-10分钟的"知识回收"）：
1. 标记项目为完成状态
2. 划除目标并反思：达成了吗？成功因素/失败原因是什么？学到了什么？
3. 重温项目文件夹，发掘可复用的半熟素材并移到其他项目/领域/资源文件夹
4. 将整个项目文件夹移入"存档"
5. 对搁置（非完成）的项目：写一条"当前状态"笔记作为海明威之桥

**项目清单的核心理念**：知识不是一次性消耗品。每次项目的终点是下次项目的起点——通过"知识回收"实现知识复利。

### 层次2：每周小结 + 月度小结（周期尺度）

**每周小结**（约30分钟，偏执行）：
1. 清理电子邮件收件箱
2. 检查日历（回顾过去+预览未来数周）
3. 清理电脑桌面
4. 清理笔记收件箱（逐条分类到PARA文件夹）
5. 确定下周的优先事项

**月度小结**（约1小时，偏反思）：
1. 评估和更新季度/年度目标
2. 评估和更新项目列表（归档完成、添加新项目）
3. 评估各责任领域是否需要改变或采取行动
4. 检视"或有任务"（如"学外语""种果园"）是否可转化为项目
5. 重新排列优先级

**两者的关系**：每周小结保持系统整洁可用（"脚"），月度小结确保方向不偏离（"眼"）。

### 层次3：处处留意（日常尺度）

不需要专门时间投入的微型维护习惯。在日常使用笔记的任何时刻，顺手做一个小改善：优化一个标题、把一条笔记从资源移到项目、合并两条相似笔记、给有共鸣的内容加粗、分享一个素材给可能受益的人。

**核心理念来源**：厨师"餐前准备"哲学——在烹饪过程中持续保持工作环境有序，而非等一切结束后再大扫除。

## A1 — 书中的应用 (Past Application)

### 案例 1：作者在苹果零售店的觉醒
- **问题**：作者在圣迭戈苹果零售店做兼职顾问时，发现客户从旧PC迁移过来的成百上千份文件杂乱无章地散落在新电脑中。他尝试逐份帮助整理，但一小时的咨询时间杯水车薪。
- **方法论的使用**：他改为将所有旧文件整体放入一个以"存档+日期"命名的文件夹中，然后聚焦客户的当前目标——"你想用这台电脑创造什么？每个人的新项目才是重点，不是历史文件的完美分类。"
- **结果**：客户们不再为混乱的历史文件焦虑，转而专注于视频制作、网站搭建、乐队录音等创作项目。那些回来的人不是要求重新整理旧文件，而是兴奋地分享创作成果。

### 案例 2：学员Fernando——肿瘤医生的知识管理
- **问题**：Fernando是全球知名医院的肿瘤学家，需要管理大量病患诊疗记录，并将培训和研究获得的知识与患者实际需求对接。
- **方法论的使用**：使用第二大脑保存每位患者的病史、治疗史和肿瘤特征摘要（"领域"文件夹）。通过定期回顾，将新学到的治疗方法与患者情况匹配。
- **结果**：能够制定更加高效的诊疗方案，将知识从培训课堂"搬运"到了病床前。

### 案例 3：学员Camille——电动车充电站设计师
- **问题**：Camille是加拿大一家初创公司的联合创始人和首席设计师，需要从大量PDF格式的研究报告中提取和整理设计相关信息。
- **方法论的使用**：将PDF研究报告导入笔记应用，对相关内容进行自由注释和评论；按项目组织设计决策相关的笔记。
- **结果**：克服了PDF格式"灵活性和易用性差"的痛点。笔记系统成为她的设计决策的"外部大脑"。

## A2 — 触发场景 (Future Trigger) ★

### 用户会在什么情境下需要这个 skill?

1. 每次开启新项目都感觉毫无章法，不知道第一步该做什么
2. 项目完成后只感到如释重负，但从不为下一次积累
3. 数字空间（笔记、桌面、邮箱、下载文件夹）越来越乱但不知从何整理
4. 感觉每天都在忙但没有方向，"忙了一周不知道自己做了什么"
5. 想要建立可持续的自我管理体系，但GTD/子弹笔记等试了一圈都不持久

### 语言信号 (用户的话里出现这些就应激活)

- "每次开启项目都乱糟糟的"
- "项目做完了但什么都没留下"
- "数字空间越来越乱"
- "感觉每周都在忙但不知道忙了什么"
- "需要定期回顾的习惯"
- "怎么让知识管理系统持续运转"
- "需要一个维护流程"
- "系统用着用着就乱了"

### 与相邻 skill 的区分

- 与 `second-brain-code` 的区别：CODE是创造知识的流程，知识生命周期管理是维护系统的流程。它们分别解决"生产"和"保养"。
- 与 `para-system` 的区别：PARA是静态的分类结构，知识生命周期是动态的维护行为——它们一起构建起知识系统的"空间"和"时间"两个维度。
- 与 `intermediate-packets` 的区别：项目完成清单的第三步（重温半熟素材）是半熟素材的"回收机制"。

## E — 可执行步骤 (Execution)

当 skill 被激活后，agent 应按以下步骤执行:

1. **诊断用户的维护痛点**
   - 询问：你的数字空间中，哪个地方最让你感到焦虑？（邮件未读数、桌面图标密度、笔记收件箱积压量、项目文件夹数量等）
   - 完成标准：确定优先解决哪个层次的问题

2. **执行一次"迷你周小结"**
   - 引导用户执行简化版：
     - 快速过一遍笔记应用的收件箱（每条约5-10秒），用PARA速查表归类
     - 确认至少有1个下周的关键优先事项
   - 完成标准：笔记收件箱清零（或显著减少）、下周优先事项已确定

3. **为下一个即将开始的项目建立"启动清单"**
   - 让用户选一个近期要启动的项目
   - 按5步走一遍（头脑风暴→浏览文件夹→搜索→集中笔记→写大纲）
   - 目标是体验"不是从零开始"的感觉
   - 完成标准：项目有了一个基于已有素材的初始大纲

4. **为上一个已完成（或搁置）的项目执行"完成清单"**
   - 让用户回想最近完成但未做"知识回收"的项目
   - 至少提取1个可复用的半熟素材放入资源/领域文件夹
   - 对未完成的项目，写一条"当前状态"笔记作为海明威之桥
   - 完成标准：至少回收了1个可复用素材

5. **设定一个可持续的维护节奏**
   - 让用户承诺一个"最低维护剂量"：
     - 每周小结的最低版本：只做"清理笔记收件箱"这一步（5分钟）
     - 月度小结的最低版本：只做"更新项目列表"（10分钟）
   - 关键原则：低频率但可持续 > 高频率但坚持不了
   - 完成标准：用户说出了一个明确的、可执行的维护频率

## B — 边界 (Boundary) ★

### 不要在以下情况使用此 skill

- 用户的数字空间极度简单（如只有不到50条笔记和10个文件）——过度结构化反而增加负担
- 用户已经有一套稳定运行且满意的个人管理习惯（GTD、子弹笔记等）——不要为了"标准化"而破坏已有的有效系统
- 紧急危机时期——当用户面临重大人生变故时，维护系统不是优先级

### 作者在书中警告的失败模式

- **脆弱完美系统**：设计一套极其复杂的维护规则——多层嵌套文件夹、精细标签分类、每日必须执行流程。一旦因出差或生病中断数日，整个系统便崩塌。"看上去很完美，实则脆弱不堪。"
- **组织完美主义**：将系统维护本身当作目的，沉迷于优化结构而不产出任何价值。警告信号：花费在调整文件夹上的时间超过使用笔记的时间。
- **"等有空再整理"的陷阱**：等整块空闲时间出现的概率几乎为零。正确做法是在日常工作中见缝插针（处处留意）。

### 作者的盲点 / 时代局限

- 清单方法假设用户有基本的时间管理能力和自律水平，对于ADHD等注意力障碍人群的适配性需要特殊考虑
- 偏向"工作=项目管理"的视角，对于艺术创作者的随性和非线性工作方式可能显得过于结构化

### 容易混淆的邻近方法论

- GTD（Getting Things Done）：GTD的每周回顾是"清空大脑+确定下一步行动"，知识生命周期的回顾是"清理数字环境+提取可复用资产+重新排列优先级"。两者互补——GTD管任务流，这个skill管知识流。
- 敏捷回顾（Sprint Retrospective）：敏捷回顾是团队层面的过程改进，知识生命周期是个人层面的系统维护。但理念相通——定期停下来审视和调整。
